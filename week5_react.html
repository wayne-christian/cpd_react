const variables can only be assigned once. It is NOT immutable (unchanging over time or unable to be changed).

             
const foo = { bar: 1 };
foo = 'bar';
// “foo” is read only.                     

//But, you can change the properties!      
const foo = { bar: 1 };
foo = 'bar';
// “foo” is read only.
            
foo.bar = 2;
console.log(foo);
// { bar: 2 }

But, what should I use?!? var? let? const?

The only difference between const and let is that const makes the contract that no rebinding will happen.

Use var for top level variables (or not at all)

Use let for localized variables in smaller scopes.

Refactor let to const only after some code has been written and you’re reasonably sure there should’t be variable reassignment.

When using var, you leak a global variable to the parent scope and the variable gets overwritten with every iteration.

Messy... hard to keep track

Arrow Functions

More concise than traditional function expressions (but you can use either)

    
// Traditional function expression.
const addNumbers = function (num1, num2) {
return num1 + num2;
}

// Arrow function expression.
const addNumbers = (num1, num2) => {
return num1 + num2;
}

Arrow Functions

        
// Arrow function without any arguments.
const sayHello = () => console.log( 'Hello!' );
            
// Arrow function with a single argument.
const sayHello = name => console.log( `Hello ${name}!` );

// Arrow function with multiple arguments. (note template lateral ${}
const sayHello = (fName, lName) => console.log( `Hello ${fName} ${lName}!` );

The Template Literal, introduced in ES6, is a new way to create a string.

    
let name = 'Wayne';

// The old way... (you may be more familiar with concatenation)

console.log('Hello, ' + name + '!'); // Hello, Wayne!
            
//ES6 template literals
console.log(`Hello, ${name}!`); // Hello, Wayne!

Within template literals you can evaluate expressions.

     
const price = 19.99;
const tax = 0.07;
const total = `The total price is ${price + (price * tax)}`;
console.log(total);
// The total price is 21.3893
// probably should round toFixed(2) method
With template literals you can more easily create multi-line strings. 
    
console.log('This is some text that flows across\two lines!');

Promises

Promises

JavaScript is single threaded, meaning that two bits of script cannot run at the same time; they have to run one after another. Typically JavaScript is in the same queue as painting, updating styles, and handling user actions (such as highlighting text and interacting with form controls). Activity in one of these things delays the others.

Promises

A promise is an object that may produce a single value some time in the future: either a resolved value, or a reason that it’s not resolved (e.g.., a network error occurred). A promise may be in one of 3 possible states: fulfilled, rejected, or pending. Promise users can attach callbacks to handle the fulfilled value or the reason for rejection.

Promises are eager, meaning that a promise will start doing whatever task you give it as soon as the promise constructor is invoked
